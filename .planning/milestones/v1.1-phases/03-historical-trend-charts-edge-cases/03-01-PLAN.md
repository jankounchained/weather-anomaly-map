## Phase Goal (user story)

**As a** person checking a location's weather, **I want to** see each of the last 7 days plotted against that day's own 30-year historical distribution — with every year's values visible and the average emphasized — plus a clear message when a day lacks enough history, **so that** I can tell at a glance how unusual the whole recent week has been, not just today.

> User story faithfully reformatted from ROADMAP.md Phase 3 Goal + Success Criteria (the roadmap states the goal in prose; the three slots here are all sourced directly from that text and CONTEXT.md's `<domain>` boundary — no new scope invented).

> Plan 01 is the dependency gate: it adds the project's first chart library, `recharts`, which every populated `TrendDayChart` in Plan 03 renders through. It ships no user-facing UI itself — its sole job is to clear the mandatory supply-chain checkpoint (recharts is flagged `[SUS]` for a "too-new version tag" false positive) and install the library cleanly, unblocking Plan 03. It runs in Wave 1 in parallel with Plan 02 (no shared files: this plan touches only package manifests, Plan 02 touches `src/`).

<objective>
Clear the package-legitimacy checkpoint for `recharts` and install `recharts@3.9.2` as a runtime dependency, so Plan 03 can compose the dot/strip trend charts. This is the first chart library in the codebase (STACK.md, 03-RESEARCH.md Standard Stack).

Purpose: 03-RESEARCH.md's Package Legitimacy Audit gives `recharts` a `[SUS]` verdict driven solely by its latest version (3.9.2) being tagged 11 days before research — a false positive on the "too-new" heuristic (10+ year package, 41M weekly downloads, canonical GitHub org, no postinstall script, ships its own types). Per protocol this still requires a blocking human-verify checkpoint before install, which cannot be auto-approved.

Output: `recharts` added to `package.json` dependencies + refreshed `package-lock.json`, a clean install and build.
</objective>
