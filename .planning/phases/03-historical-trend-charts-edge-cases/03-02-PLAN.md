## Phase Goal (user story)

**As a** person checking a location's weather, **I want to** see each of the last 7 days plotted against that day's own 30-year historical distribution — with every year's values visible and the average emphasized — plus a clear message when a day lacks enough history, **so that** I can tell at a glance how unusual the whole recent week has been, not just today.

> Plan 02 is the data + statistics slice. It delivers a real, user-visible behavior change on its own — today's anomaly card now applies the stricter "at least half the years" usable-data rule (D-09/D-10), so a location with sparse history shows the existing "Couldn't compute an anomaly here." state instead of a misleadingly-confident reading — AND it pipes the 7 recent daily values + the per-day trend computation that Plan 03 renders. It runs in Wave 1 alongside Plan 01 (no shared files). The chart pixels land in Plan 03; the correctness that makes those charts trustworthy lands here and is fully unit-tested.

<objective>
Build the shared usable-data threshold and the 7-day data pipeline (VIZ-01, ROBU-01): (1) a single `hasUsableSampleCount` helper that both today's anomaly and every trend day gate on (D-09/D-10), (2) a `computeTrendDay` function returning the per-day chart inputs (samples ∪ mean ∪ actual) or an unusable marker (D-11/D-14), and (3) an extended `getCurrentWeather`/`useCurrentWeather` that fetches all 7 recent daily means in the existing forecast call (D-13).

Purpose: D-10 requires exactly ONE definition of "usable history" across the whole app — retrofitting `computeAnomalyForToday`'s old `samples.length < 2` gate to the shared helper is in-scope robustness tightening, not scope creep. Sourcing the 7 actual values from the forecast API (not the archive/reanalysis endpoint) sidesteps the archive's few-day lag (03-RESEARCH.md Pattern 1, verified live).

Output: extended `src/anomaly/{anomaly,types,anomaly.test}.ts` and `src/weather/{client,types,useCurrentWeather,client.test}.ts`. No component/UI changes in this plan (Plan 03 owns rendering). `AnomalyCard.tsx` is intentionally NOT modified — the stricter gate flows into it transitively (its existing `anomaly === null` error branch already handles the newly-stricter null case; 03-PATTERNS.md confirms no edit needed there).
</objective>
